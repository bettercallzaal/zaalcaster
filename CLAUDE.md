## Create a tokenless empire (2026-07-14; NOW ON THE EMPIRE TAB, custom mode only - see 2026-07-15 entry)
Empire Builder's create call is real and documented (found via their gitbook, not guessed): `POST /api/deploy-empire-tokenless`, header `x-api-key`, body `{mode: 'custom'|'farcaster', owner, name, bio?, logoUri?, fid?, farcasterUsername?, signature, message}` - `signature` is an EIP-191 `personal_sign` the `owner` wallet must produce over an exact required `message` string (`I am deploying a custom tokenless Empire named X` / the farcaster-mode variant with fid). This is why zaalcaster CANNOT deploy an empire server-side alone - it has no private key and never will. The split: `empire.js:deployTokenlessEmpire()` + `tokenlessEmpireMessage()` (server relay + the canonical message builder, EMPIRE_BUILDER_API_KEY from env or creds file, mirrors juke.js's key-loading pattern) vs `public/index.html`'s Grow tab "+ create a tokenless empire" card (wallet connect - tries the Farcaster mini-app wallet `window.__fcsdk.wallet.ethProvider` first, falls back to `window.ethereum` - then `personal_sign`, then POSTs the signed payload to `/api/stats` POST). `api/stats.js` re-derives the expected message server-side from `mode`/`name`/`fid` and rejects anything that doesn't match, so a tampered client can't smuggle a different signed message through. Zaal-only (blockedByAuth, same as every other write route) - the Grow tab itself is already hidden from SIWF guests. Farcaster mode is flagged with an explicit warning (research doc 1092, Adrian's live warning: attaching to a profile is permanent, can't create a second one for that profile ever). No new npm dependency - wallet connection is native `window.ethereum` / the already-loaded mini-app SDK, no WalletConnect yet (would need one, deferred). UNPROVEN: the actual click-connect-sign-deploy flow in a real browser with a real wallet is Zaal's live test - verified everything short of that (message format matches Adrian's spec exactly, full server-side auth/validation chain live-tested with real session cookies, safe no-op when the key or a real network call isn't available).

## Sign In With Farcaster (2026-07-14) - replaces the password gate
Zaal asked to let other Farcaster users sign in and see a read-only slice, instead of a shared password. Rebuilt auth.js + api/auth.js around Neynar's SIWN (Sign In With Neynar), Zaal's pick over Farcaster's own AuthKit (would've added a new npm dep) or the raw relay protocol (more code to own) - SIWN needed zero new deps and reuses the existing NEYNAR_API_KEY.
- **Roles, not a single gate**: any real Farcaster user can sign in. fid === USER_FID/ZAAL_FID/config.fid (19640) -> role 'zaal' (full read+write, unchanged). Anyone else -> role 'guest' (read-only, public-surface-only). Zaal's own scope pick for guests: Feed, Channels, Search, Empire reads. NOT guest-visible: Inbox, Post/Compose, Daily, Grow, and any write action anywhere.
- **Session model**: cookie is `${fid}.${issuedAt}` HMAC-SHA256-signed with SESSION_SECRET - NOT the SIWN signer_uuid (that's write-capable for its owner and never leaves the server). `getSession(req)` verifies the signature + 30d expiry and returns `{fid, role}` or null. Live-verified: a tampered cookie (swapped fid, same signature format) is correctly rejected; only a genuinely re-signed cookie passes.
- **Never trust the client**: api/auth.js POST receives `{fid, signer_uuid}` from the browser's SIWN callback, then independently re-verifies via `lib.js:getSignerInfo()` (GET /farcaster/signer?signer_uuid=, same call getPostingHealth already used for Zaal's own signer) - only issues the session cookie if Neynar confirms `status:'approved'` AND the returned fid matches the claim. Live-verified end to end with Zaal's real signer_uuid (200, role 'zaal', cookie set) and a bogus uuid (401, no cookie set).
- **Two guards, same shape as before**: `blockedByAuth` (Zaal-only - inbox/send/react/digest/state/stats, all UNCHANGED, zero edits needed since the function name+contract stayed the same) and new `blockedByGuestAuth` (any signed-in user - feed/search/view). feed.js is split further: trending/frames/empire/following/trending/foryou/channel are guest-OK, but myactivity/notifs/bestfriends/type=list stay Zaal-only via an inline `blockedByOwner` check - live-verified each branch (403 for guest on owner-only modes, 200 with real data on public modes, 401 with no session).
- **Frontend**: password input replaced with the official SIWN button snippet (`neynar_signin` div + `data-client_id` + their hosted script), injected client-side only once `/api/auth` GET returns a configured `neynarClientId` (env `NEYNAR_CLIENT_ID`, public value, not secret). `ROLE` global drives `applyRole()` (hides Daily/Inbox/Post/Grow tab buttons + the feedback FAB for guests) and a `switchTab()` guard (redirects to Feed if a guest reaches a restricted tab via cmd+k or a stale UI path) - server-side gating is the real boundary, this is UX.
- **Gate off** (SESSION_SECRET unset) behaves exactly like the old APP_PASSWORD-unset case: every request treated as Zaal, unchanged for local CLI / Vercel-auth-only deploys.
- TWO MANUAL STEPS (Zaal): register an app in the Neynar dev portal (Settings tab) to get NEYNAR_CLIENT_ID, and set both NEYNAR_CLIENT_ID + a random SESSION_SECRET in Vercel. Until NEYNAR_CLIENT_ID is set, the gate shows "sign-in not configured yet" instead of a broken button.

## Reliability trio (2026-07-06, from Iman's ideas PDF)
- Rate-limit guard: fetchNeynar retries 429 with Retry-After/backoff (3 tries) so a busy day / live Space degrades gracefully.
- Action ledger: logAction() appends every write (cast/reply/quote/like/recast/delete) to KV list zc:actions (best-effort, never blocks a post); postCast/postReaction/deleteCast call it; read with `node bin/actions.js [N]`.
- Self-monitor: cron/post-due writes zc:cron:last each run; getSystemHealth() checks posting-signer health + cron freshness; `node bin/health.js` prints it and exits non-zero if unhealthy (catches the two past silent failures: deploy-rate-limit killing cron, signer/key mismatch).
- bin/feedback.js (in-app feedback reader) + bin/health.js + bin/actions.js all load KV creds from ~/.zao/private/farcaster-zaal.env (Zaal added KV_REST_API_URL/_TOKEN there).
